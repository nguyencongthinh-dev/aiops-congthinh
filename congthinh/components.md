# A2: Component-Decision Table

This table outlines the selected component or vendor for each required capability, justifying the choice and acknowledging the risk of changing direction in six months.

| Capability | Chosen Component / Vendor | Why this one? (Justification) | What gets worse if we change our mind in six months? |
| :--- | :--- | :--- | :--- |
| **Metrics ingestion + storage + query** | **VictoriaMetrics** (OSS, self-hosted) | Handles high-cardinality custom metrics efficiently at a fraction of Datadog's cost, avoiding penalty fees while natively supporting PromQL for Grafana integration. | Migrating away from PromQL-based alert rules and dashboards to a proprietary syntax (like Datadog's) would require rewriting the entire monitoring logic manually. |
| **Logs ingestion + storage + search** | **Vector** (Ingest/Route) + **Redpanda** (Buffer) + **Grafana Loki** (Storage/Search) | Loki drastically cuts storage costs by avoiding full-text indexing (backing data to S3), while Vector filters/routes logs and Redpanda buffers them to ensure zero data loss. | The operational team would have to endure another major migration of LogQL queries and re-train engineers on a new logging syntax. |
| **Distributed tracing** | **Grafana Tempo** (OSS, self-hosted) | S3-backed storage makes it cheap enough to enable 100% tail-based sampling for errors, solving the current 1% sampling blindness to tail latency. | We would lose the tight integration between TraceQL and Loki/Prometheus in Grafana, breaking the seamless "Metrics to Traces" correlation workflow. |
| **Alerting (rule engine) + correlation / grouping** | **Prometheus Alertmanager** (OSS, self-hosted) | Provides powerful label-based grouping and inhibition rules, effectively deduplicating alert storms before they hit PagerDuty. | Alert grouping logic would have to be rebuilt in an external SaaS platform (like PagerDuty Event Intelligence), incurring massive premium licensing costs. |
| **Incident routing + paging** | **PagerDuty Business** (SaaS) | It works well today, and replacing the final paging surface saves very little money while introducing huge risk to incident response reliability. | We would have to recreate and test all the escalation policies and user schedules in a new system, risking a missed page during cut-over. |
| **Dashboards + SLO tracking** | **Grafana Cloud Pro** (SaaS) | Acts as the ultimate "Single Pane of Glass," consolidating Metrics, Logs, and Traces into one UI to eliminate context switching and reduce MTTR. | We would break the "single pane of glass" paradigm, forcing engineers to once again log into multiple fragmented tools during critical incidents. |
| **Security Audit Logging** *(Added Value)* | **AWS S3 + AWS Athena** (SaaS) | Satisfies the compliance team's need for long-term historical log queries using standard SQL at pennies per GB, bypassing the need for Splunk. | We would have to migrate terabytes of Parquet files back into an expensive indexed database, facing massive data transfer fees. |
| **AIOps & Anomaly Detection** | **VictoriaMetrics `vmanomaly`** (OSS, self-hosted) + **Grafana Machine Learning** (SaaS) | Provides dynamic, machine learning-driven anomaly detection to prevent alert fatigue, natively querying data from VictoriaMetrics and surfacing outliers in Grafana Cloud. | We would lose custom ML model training history and configurations, forcing us to rebuild dynamic thresholds and revert to noisy static threshold alerts. |
| **Log Collection & Routing** | **Vector** (OSS, self-hosted agent) | Written in Rust, memory-safe, extremely low footprint (~30MB RAM), supporting high-throughput log parsing, dynamic routing, and disk-backed safety queues. | Migrating back to OTel Collector for logs would increase memory consumption on service nodes by 5-10x and lose specialized Vector log-routing pipelines. |
| **Telemetry Ingestion Buffer** | **Redpanda** (OSS, self-hosted cluster) | Kafka-compatible, written in C++ with thread-per-core layout, providing sub-millisecond tail latency buffering for logs/metrics without JVM tuning overhead. | Bypassing the buffer would directly expose Loki and VictoriaMetrics to log/metric storms, risking system-wide outages and telemetry loss during storage database maintenance windows. |
| **Auto-Instrumentation & APM** | **Grafana Beyla** (OSS, self-hosted eBPF agent) | Uses eBPF at the kernel level to capture network HTTP/gRPC metrics and traces automatically, eliminating the need to modify application source code or build/deploy cycles. | Reverting to manual SDK-based instrumentation would require weeks of development time across 10 service codebases, delaying tracing coverage and increasing code complexity. |
| **Incident Remediation Engine** | **Keep** (OSS, self-hosted) | Standardized alerting action platform that intercepts Alertmanager webhooks and triggers automated remediation scripts (restarts, clearing disk) before escalations. | Decommissioning Keep forces us to route all alerts directly to PagerDuty, exposing on-call engineers to alarm fatigue from easily resolvable issues. |
